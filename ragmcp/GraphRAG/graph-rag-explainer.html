What is Graph RAG?

Retrieval-Augmented Generation (RAG) traditionally relies on vector embeddings to match user queries with relevant document chunks. While effective for semantic similarity, classical RAG struggles significantly with multi-hop reasoning — questions requiring connections across disjointed pieces of information. Graph RAG solves this by structuring data into a knowledge graph, enabling explicit relationship traversal rather than mere semantic matching.

⚠️ The Classical RAG Problem Consider this three-sentence corpus and the question below. 
# The dataset
texts = [
    "Albert Einstein developed the theory of relativity and influenced quantum mechanics.",
    "The Manhattan Project was a secret U.S. project during WWII that developed the atomic bomb.",
    "The Manhattan Project researched topics like the theory of relativity and quantum mechanics."
]

question = "How did Albert Einstein contribute to the atomic bomb?"

In the classical workflow, the system embeds the question and retrieves top-k chunks by cosine similarity. The top chunks mention Einstein and relativity, or separately mention the atomic bomb — but no single chunk directly connects them. The semantic distance between "Einstein" and "atomic bomb" is large without intermediate context. The LLM receives fragmented evidence and either hallucinates or says it doesn't know.

🔗 How Graph RAG bridges the gap Graph RAG extracts structured triples from the corpus and builds a traversable knowledge graph. 
# Triples extracted by LLM during indexing
(Albert Einstein,     developed,          theory of relativity)
(Albert Einstein,     influenced,         quantum mechanics)
(Manhattan Project,   researched,         theory of relativity)
(Manhattan Project,   researched,         quantum mechanics)
(Manhattan Project,   developed,          atomic bomb)
(Manhattan Project,   was,                secret U.S. project during WWII)

When the question arrives, Graph RAG detects the anchor entity Albert Einstein and traverses the graph to depth 2:

Albert Einstein
→
developed
→
theory of relativity
→
researched by
→
Manhattan Project
→
developed
→
atomic bomb

The multi-hop path Einstein → theory of relativity → Manhattan Project → atomic bomb is explicitly surfaced. The LLM now receives the complete logical chain as structured context and can answer confidently without any hallucination. This is what Graph RAG was built for.

✅ Key insight Graph RAG doesn't search for similarity — it navigates logical structure. Each edge is a verified fact from the corpus, not a statistical proximity score.

Why Domain-Specific Keywords Are a Great Plus in Graph RAG

In Graph RAG systems, anchor detection is the critical step of identifying the correct entity in the knowledge graph to initiate traversal. Generic NLP often fails here, extracting broad nouns that dilute retrieval precision. Domain-specific keywords serve as high-fidelity anchors, ensuring the system navigates directly to relevant subgraphs rather than drowning in semantic noise.

Consider the entity radioactivity research derived from Marie Curie's work. As an anchor, this term links exclusively to Curie, polonium, and radium. By contrast, a generic anchor like research connects to every scientist in the dataset, producing a massive, unfocused subgraph. Specificity dictates accuracy.

🔍 Comparison: Generic vs. Domain-Specific Keywords

User Query Generic Keyword Domain-Specific Keyword Correct Anchor
"Who advanced our understanding of radioactive materials?" materials → noise radioactivity research ✓ Marie Curie
"Who challenged the geocentric model?" model → software? ✗ heliocentrism ✓ Galileo Galilei
"Who developed AC power?" power → political? ✗ alternating current (AC) electricity systems ✓ Nikola Tesla
"What theory did Einstein publish?" theory → philosophy? ✗ theory of relativity ✓ Albert Einstein
"Who programmed the Analytical Engine?" engine → locomotive? ✗ Analytical Engine ✓ Ada Lovelace

💡 Domain Keywords Reduce Ambiguity

Domain-specific terms dramatically reduce polysemy. In a general corpus, "relativity" might refer to social theories or philosophical concepts. The specific phrase theory of relativity is unambiguous — it maps directly to Einstein and physics-related nodes. Similarly, alternating current (AC) electricity systems is unambiguous, while "electricity" alone would match Tesla, Edison, Faraday, and dozens of other nodes.

# This is the known_entities.json used by the notebook for anchor matching
known_entities = [

    "Analytical Engine",    # ← compound noun preserved
    "Apollo 11", "Catholic Church", "Charles Babbage",
    "Galileo Galilei", "Isaac Newton", "Manhattan Project",
    "Marie Curie", "Nikola Tesla", "atomic bomb",
    "alternating current (AC) electricity systems",  # ← full domain term
    "heliocentrism",            # ← specific concept, not "model"
    "polonium", "quantum mechanics", "radioactivity research",
    "radium", "the theory of relativity"
]
🧠 How domain keywords are built They come from two sources: (1) pre-built entity lists maintained by domain experts (like known_entities.json), or (2) LLM-extracted entities during indexing, where the LLM is explicitly instructed to preserve compound nouns and full-phrase concepts rather than splitting them into generic tokens. The notebook uses both: a seeded entity list plus LLM discovery during triple extraction.

How Semantic Embeddings Improve Keyword Search in Graph RAG

Exact keyword matching is inherently brittle within knowledge graphs. Consider a user asking about "nuclear weapons research" — the graph contains nodes for atomic bomb and Manhattan Project, but no node labeled "nuclear weapons research" exists. Exact match fails silently. Similarly, a query about "radioactive elements" will not match stored entities like polonium, radium, or radioactivity research.

⚠️ The exact-match trap The user speaks in their vocabulary. The graph is built in the corpus vocabulary. These never fully align. Bridging that gap is the job of embeddings.

🔢 Embedding-Based Anchor Detection

The solution: embed the query and all known graph entities into the same vector space. Find top-k entity candidates by cosine similarity. Even when wording diverges, semantically related concepts cluster together.

# Conceptual example — embedding similarity scores
query = "nuclear weapons"
top_k_entities = [
    ("atomic bomb",                  0.89),   # ← strong match
    ("Manhattan Project",              0.81),   # ← strong match
    ("World War II",                   0.72),   # ← contextual match
    ("the theory of relativity",        0.41),   # ← weak, skip
]

query = "radioactive elements"
top_k_entities = [
    ("radioactivity research",          0.91),   # ← strong match
    ("radium",                          0.85),   # ← strong match
    ("polonium",                        0.83),   # ← strong match
    ("Marie Curie",                     0.74),   # ← contextual match
]

📉 Impact on Traversal

QueryExact Match ResultEmbedding Match Result
"nuclear weapons" No node found → traversal fails ✗ Anchor: "atomic bomb" → Manhattan Project → Einstein ✓
"radioactive elements" No node found → empty context ✗ Anchor: "radioactivity research" → Marie Curie → polonium, radium ✓
"laws of physics" No node found → fails ✗ Anchor: "theory of relativity" → Einstein; "quantum mechanics" → Manhattan Project ✓

⚙️ The Embedding-Enhanced Anchor Algorithm

def find_anchor_entities(query: str, known_entities: list, top_k: 3) -> list:
    # 1. Embed the query
    q_vec = embed(query)

    # 2. Embed all known graph entities (cache these — they don't change)
    entity_vecs = {e: embed(e) for e in known_entities}

    # 3. Rank by cosine similarity
    scores = [(e, cosine_sim(q_vec, v)) for e, v in entity_vecs.items()]
    scores.sort(key=lambda x: -x[1])

    # 4. Return top-k entity candidates as anchor seeds
    return [e for e, _ in scores[:top_k]]

def retrieve_with_graph(query: str, G: nx.Graph, depth: 2) -> list:
    anchors = find_anchor_entities(query, list(G.nodes), top_k=3)
    context = []
    for anchor in anchors:
        subgraph = search_graph(anchor, depth=depth)
        context += [f"{u} {d['relation']} {v}"
                    for u, v, d in subgraph.edges(data=True)]
    return context  # pass to LLM
🧠 Embeddings as the "entry point navigator" The graph's value is entirely in its traversal capability — but you can only exploit it if you land on the right starting node. Semantic embeddings are the navigation layer that bridges natural language queries to structured graph topology. Without them, graph traversal is only reachable via exact terminology.

Graph RAG + Classical Embeddings — Two-Phase Retrieval

The most powerful RAG architecture combines both approaches in a two-phase retrieval pipeline. Pure embeddings excel at semantic similarity but fail at multi-hop reasoning. Pure graph retrieval offers rigorous reasoning chains but fails when query vocabulary diverges from graph terminology. Together, they cover each other's blind spots.

Phase 1 Semantic Entry — Classical Embeddings

Embed the user query. Search entity embedding index for top-k anchor candidates (semantic similarity). Optionally also search raw text chunk index for directly relevant passages. Output: ranked anchor candidates + optional direct context snippets.

Phase 2 Graph Expansion — Knowledge Graph Traversal

For each anchor candidate: traverse graph to depth d. Collect all (subject, relation, object) triples in the subgraph. These triples form structured factual context that vector search alone can never produce.

Phase 3 Context Fusion + LLM Generation

Merge Phase 1 direct passages + Phase 2 graph triples into a unified context window. LLM generates an answer grounded in both unstructured passages AND verified structured facts, significantly reducing hallucination risk.

🔬 Worked Example

Query: "What scientific concepts connected Einstein to wartime research?"

# Phase 1 — Semantic search finds anchor candidates
anchors_phase1 = [
    "Albert Einstein",       # sim: 0.88
    "theory of relativity",  # sim: 0.84
    "quantum mechanics",     # sim: 0.79
]

# Phase 2 — Graph traversal from "Albert Einstein" (depth=2)
graph_triples = [
    ("Albert Einstein",    "developed",    "theory of relativity"),
    ("Albert Einstein",    "influenced",   "quantum mechanics"),
    ("Manhattan Project",  "researched",   "theory of relativity"),
    ("Manhattan Project",  "researched",   "quantum mechanics"),
    ("Manhattan Project",  "developed",    "atomic bomb"),
    ("Manhattan Project",  "was",          "secret U.S. project during World War II"),
]

# Phase 3 — Fused context sent to LLM
context = direct_passages + [f"{s} {r} {o}." for s, r, o in graph_triples]
# → LLM produces high-quality, multi-hop answer with zero hallucination

⚖️ When to Use Each Approach

ScenarioBest ApproachReason
Simple factual lookup, low latency required Pure Embeddings Fast, single-hop, no graph needed
Structured domain, standardized terminology Pure Graph RAG Exact matches reliable, relationships explicit
Complex analytical questions, multi-entity Hybrid (Phase 1+2+3) ✓ Semantic entry + logical expansion = maximum accuracy
Paraphrased queries in specialized domains Hybrid (Phase 1+2+3) ✓ Embeddings bridge vocabulary gap; graph bridges reasoning gap
✅ The bottom line Classical embeddings handle semantic flexibility — they bridge vocabulary gaps and find related concepts. Graph traversal handles logical rigor — it navigates verified relationship chains that vector distances don't capture. Together they form a retrieval system that is both linguistically robust and factually precise. Neither is sufficient alone for complex, real-world queries.

5 · On-the-Fly Dependency Discovery via Multiple Queries

What if you don't have a pre-built knowledge graph at all? One lightweight alternative is to discover dependencies dynamically by issuing multiple sequential LLM or retrieval queries at runtime, each one informed by the answer to the previous one. No graph indexing step, no triple extraction pipeline — just a loop.

🔄 How It Works

Step 1 Anchor Query — find the entry point
# Q1 — entry-point question
q1 = "What did Einstein contribute to nuclear physics?"
a1 = llm.query(vectorstore, q1)
# a1 → "Special relativity showed mass-energy equivalence (E=mc²)"
Step 2 Dependency Extraction — ask what concepts this answer depends on
# Extract entities / dependencies from a1
deps_prompt = f"List the key concepts mentioned in: '{a1}' as a comma-separated list."
deps = llm.complete(deps_prompt)
# deps → "special relativity, mass-energy equivalence, E=mc²"
Step 3 Chain — query each dependency, build a context window
context_parts = [a1]
for dep in deps.split(","):
    follow_up = f"How did {dep.strip()} connect to the Manhattan Project?"
    context_parts.append(llm.query(vectorstore, follow_up))

# Final synthesis — all gathered context in one prompt
final_answer = llm.complete("Using this context: " + "\n".join(context_parts)
                            + "\n\nExplain how Einstein's work led to atomic weapons.")

⚖️ On-the-Fly Discovery vs. Real Graph RAG

DimensionOn-the-Fly (Multi-Query)Real Graph RAG
Setup cost Zero — no indexing ✓ High — extract triples, build graph
Query latency High — N sequential LLM calls Low — single graph traversal ✓
Depth of reasoning Shallow (2–4 hops before drift) Deep — 10+ hops reliably ✓
Reproducibility Low — LLM picks different deps each run High — graph is deterministic ✓
Hallucination risk High — errors compound per hop Low — traversal stays on verified edges ✓
Works without pre-built corpus Yes — pure LLM ✓ No — requires structured graph store
Scales to large corpora Poor — costs grow linearly with hops Excellent — graph index is sublinear ✓
✅ When on-the-fly discovery is the right choice
  • Prototype / exploration — you need quick multi-hop answers before committing to graph indexing.
  • Small corpora (< 500 documents) — overhead of graph extraction isn't justified.
  • One-off queries — the same topic won't be queried repeatedly, so a persistent graph adds no value.
  • Dynamic content — documents change so frequently that maintaining a consistent graph is impractical.
⚠️ When to do real Graph RAG instead
  • Production accuracy requirements — compounding LLM errors over multi-hop chains is unacceptable.
  • Deep reasoning (5+ hops) — chain-of-queries drifts badly beyond 3–4 steps.
  • High query volume — paying N×LLM-cost per query destroys economics; graph traversal is near-free after indexing.
  • Auditable reasoning — the graph path is a literal citation trail; multi-query chains are not.
  • Domain with stable ontology — medical, legal, scientific — where relationships are verifiable facts.
💡 The practical rule of thumb Multi-query chaining is a proof-of-concept tool. If you find yourself needing 3+ hops regularly, or accuracy matters, graduate to a real graph store. Think of on-the-fly discovery as scaffolding — useful while you're building, replaced once you're serious.
