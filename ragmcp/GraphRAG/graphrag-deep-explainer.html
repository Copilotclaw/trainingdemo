🧠 1. What is GraphRAG?

Before diving into the notebook, let's understand the concept — because it's actually quite clever.

🔵 Key Concept: Plain RAG vs GraphRAG

Standard RAG (Retrieval Augmented Generation) works like a smart search engine: you ask a question, it finds the most similar text chunks, and hands them to an LLM to summarize. Simple but shallow.

GraphRAG goes deeper. Instead of just finding similar text, it first builds a map of knowledge — who is connected to whom, what causes what, who did what. Then when you ask a question, it navigates that map to pull the relevant relationships, not just random matching sentences.

A concrete example:

Imagine you ask: "How did Einstein's work connect to the atomic bomb?"

  • Plain RAG might find: a sentence about Einstein, a sentence about the bomb. Maybe. Depends on keyword overlap.
  • GraphRAG finds: Einstein → influenced → Quantum Mechanics → used by → Manhattan Project → produced → Atomic Bomb. It traces the chain.
💡 Why graphs?

Graphs are how humans actually store knowledge in their heads. "Who was at the party?" isn't a keyword problem — it's a network traversal. Graphs let computers reason the same way.

🏗️ 2. The Full Pipeline

The notebook implements a complete GraphRAG pipeline from raw text to answered question. Here's how data flows through it:

1 Raw Text (Historical facts)
11 sentences about Einstein, Curie, Newton, etc. → split into Document chunks
↓
2 LLM-based Entity Extraction
qwen2:0.5b reads each chunk → outputs (Subject, Predicate, Object) triples
↓
3 Graph Construction (NetworkX)
Each triple becomes: Node(Subject) → Edge(Predicate) → Node(Object) in a directed multigraph
↓
4 Question → Anchor Entity
User asks a question → word-overlap scoring finds the best matching graph node
↓
5 Subgraph Retrieval (BFS)
BFS from anchor node, depth=2 → collects all nearby nodes and their connections
↓
6 LLM Answer Synthesis
Subgraph edges formatted as context → LLM answers question using those facts

🔵 What is Ollama?

Ollama is like a "local model server" — it runs AI models on your own machine (or Colab VM) without sending anything to the cloud. Think of it as running your own private ChatGPT, completely offline.

It listens on http://localhost:11434 — an HTTP server that accepts AI chat requests. The ollama serve command starts that server.

🔵 What is qwen2:0.5b?

qwen2 is a model family from Alibaba Cloud. The 0.5b means 0.5 billion parameters — that's the "size" of the model. For reference:

  • qwen2:0.5b → ~350MB on disk. Runs on CPU. Fast but limited reasoning.
  • GPT-4 → estimated 1.8 trillion parameters. Cloud only. Expensive.

Using 0.5b means the notebook runs free on Colab's CPU, but at the cost of output quality — which is why some results look messy later.

⚠️ Why zstd?

Ollama's installer packages its binary as a .tar.zst file (zstandard compression). Google Colab doesn't have zstd installed by default, so extraction fails without it. That first line fixes that dependency before downloading anything else.

This creates the "LLM object" — a Python handle to your local AI model. When code later calls llm.invoke("some text"), it sends an HTTP request to Ollama on port 11434, gets a response, and returns it.

🔵 What is temperature?

Temperature controls how "random" the model's responses are:

  • 0.0 — fully deterministic. Same input always gives same output. Useful for structured tasks (JSON extraction, code).
  • 0.7 — balanced. Creative enough for natural language, focused enough for tasks.
  • 1.0+ — very random, creative, sometimes incoherent. Good for brainstorming.

For this notebook, 0.7 is reasonable but slightly too high for the extraction task in cell 14 — lower temperature (0.1–0.2) would give more consistent triple formatting.

📄 4. Data Loading & Chunking (Cells 9–10)

Cell 10 — Historical Text Data

In [10] code · no output
historical_texts = [ "Albert Einstein developed the theory of relativity...", "Marie Curie discovered polonium and radium...", "The Manhattan Project was a secret U.S. project...", # ... 8 more sentences ] docs = [Document(page_content=text) for text in historical_texts] splitter = RecursiveCharacterTextSplitter(chunk_size=500, chunk_overlap=50) documents = splitter.split_documents(docs)
🔵 What is a LangChain Document?

Document(page_content=text) is a simple wrapper object. It holds the text content plus optional metadata (like source file, page number). LangChain functions expect this format rather than plain strings.

🔵 What is Text Chunking?

LLMs have a limited "context window" — they can only process a certain amount of text at once. For large documents (books, reports, wikis), you must split them into chunks first.

chunk_size=500 means each piece is at most 500 characters.

chunk_overlap=50 is the clever part: each chunk overlaps by 50 characters with the previous one. This prevents information loss when a sentence is cut in the middle.

Example without overlap:

Chunk 1: "...Einstein influenced the development of quantum" Chunk 2: "mechanics, which was later used in the Manhattan..." ← "quantum mechanics" is split across chunks! Context lost.

With overlap=50:

Chunk 1: "...Einstein influenced the development of quantum" Chunk 2: "of quantum mechanics, which was later used..." ← "quantum" appears in both chunks. Context preserved.
🔵 What is RecursiveCharacterTextSplitter?

This splitter is smarter than a plain character split. It tries to split at natural boundaries in this order: \n\n (paragraphs) → \n (lines) → . (sentences) → (words) → characters. It only goes to the next level if the chunk is still too long. This keeps semantic units intact where possible.

Since the 11 sentences are each under 500 characters, chunking doesn't actually split them further here — you end up with roughly 11 documents. The splitter is included to make the notebook work for larger inputs too.

🏗️ 5. The Extraction Prompt (Cell 12)

In [12] code · no output (defines the extraction chain)
dynamic_entity_prompt = PromptTemplate.from_template(""" You are an expert in information extraction. Extract factual triples and new entities from the following text. - Return the triples as plain lines in the format: Subject, Predicate, Object - Do not use dashes, bullets, or numbering. - Return the new entities as a plain list, one per line. Text: {input} Known Entities: {known} Respond in this format exactly: Triples: Subject1, Predicate1, Object1 Subject2, Predicate2, Object2 New Entities: Entity1 Entity2 """) dynamic_entity_chain = dynamic_entity_prompt | llm | StrOutputParser()
🔵 What are Triples (Subject, Predicate, Object)?

A triple is the fundamental building block of a knowledge graph. It represents one fact as three parts:

  • Subject — the thing the fact is about → Albert Einstein
  • Predicate — the relationship → developed
  • Object — what it's connected to → Theory of Relativity

In graph terms: Subject and Object become nodes. Predicate becomes the labeled edge (arrow) between them.

🔵 What is LangChain's pipe syntax |?

LangChain uses a "chain" concept. The | (pipe) operator chains components together into a pipeline that runs left to right:

prompt | llm | StrOutputParser() ↑ ↑ ↑ Format Send to Extract input Ollama text from text model response

When you call dynamic_entity_chain.invoke({"input": text, "known": entities}), it: fills the template → sends to LLM → extracts the string response. This is called LCEL (LangChain Expression Language).

🔵 What is StrOutputParser?

LLM responses come back as complex objects (with metadata, token counts, etc.). StrOutputParser extracts just the text content — the part you actually want to read and parse.

⚠️ The dangerous part of this prompt

See those example placeholder names in the prompt?

Triples: Subject1, Predicate1, Object1 Subject2, Predicate2, Object2

They're there to show the format, not actual data. But qwen2:0.5b doesn't always understand that distinction well. Sometimes it copies those placeholder names as actual values — which is exactly what happens in Cell 14 for document 2.

🌐 6. Building the Knowledge Graph (Cell 14)

This is the most important — and most revealing — cell in the notebook. Everything goes through the LLM here, and the quality of the output depends entirely on how well that tiny model follows instructions.

In [14] code · long output · ~3–5 minutes
# === CONFIG === REINDEX = True # True = rebuild from LLM. False = load saved files. GRAPH_PATH = "historical_graph.json" ENTITIES_PATH = "known_entities.json" # NetworkX MultiDiGraph — Multiple edges allowed, directed arrows G = nx.MultiDiGraph() entity_usage = defaultdict(int) known_entities = set() # For each document chunk... for i, doc in enumerate(documents): # Ask LLM to extract triples from this chunk output = dynamic_entity_chain.invoke({"input": doc.page_content, "known": known_str}) # Parse: split on "New Entities:" to separate the two sections sections = output.strip().split("New Entities:") triples_block = sections[0].replace("Triples:", "").strip() # For each triple line, split on comma → (S, P, O) for line in triples_block.split("\n"): parts = [p.strip() for p in line.split(",")] if len(parts) == 3: s, r, o = parts G.add_node(s) G.add_node(o) G.add_edge(s, o, relation=r) # Add the edge with the predicate as label entity_usage[s] += 1 entity_usage[o] += 1 # Save to disk for caching (REINDEX=False next time = loads instantly) json.dump(json_graph.node_link_data(G), f)
🔵 What is a MultiDiGraph?

NetworkX offers several graph types:

  • Graph — undirected, one edge per pair of nodes
  • DiGraph — directed (arrows), one edge per direction
  • MultiDiGraph — directed, multiple edges allowed between the same two nodes

MultiDiGraph is used here because the same two entities might have several different relationships. E.g., Einstein might "developed" AND "influenced" Quantum Mechanics — two different edges between the same nodes.

🔵 What is entity_usage?

A dictionary that counts how many times each entity appears across all extracted triples. Entities mentioned often (Einstein, Curie) get higher scores. This count is used later to prioritize which entities to include in the next extraction prompt (only the top 40 are passed as "known entities"), and as a tiebreaker in anchor detection.

🔵 What is REINDEX and the caching system?

Building the graph requires calling the LLM for every document chunk — slow and uses compute. The cache works like this:

  • REINDEX=True → runs all LLM calls, rebuilds graph from scratch, saves to historical_graph.json and known_entities.json
  • REINDEX=False → loads saved files directly, skips all LLM calls (instant!)

This is a practical optimization: once you've built the graph once, you don't need to redo it every time you restart the notebook — unless the source data changes.

🔍 7. Search Logic (Cells 15–16)

Cell 15 — Directed Search (alternative, not used in main flow)

In [15] code · no output (function definition)

search_graph_directed does BFS (Breadth-First Search) on the directed version of the graph — it follows arrows in their defined direction only.

🔵 What is BFS (Breadth-First Search)?

BFS is a classic graph traversal algorithm. Imagine you're standing at a node (your anchor entity). BFS explores all neighbors first (depth 1), then all neighbors of neighbors (depth 2), etc.

Visual example for anchor = Einstein, depth = 2:

Depth 0: [Einstein] Depth 1: [Theory of Relativity, Quantum Mechanics, Manhattan Project] Depth 2: [Atomic Bomb, Physics, World War II, ...] All these nodes become your "subgraph" context.

Setting depth=2 keeps the context focused. At depth 5 you might end up with half the graph — too much noise for the LLM to process.

This directed version isn't used in the main pipeline because with messy extracted data, many connections point the "wrong" direction. The undirected version in Cell 16 is more resilient.

Cell 16 — The Main Search Function (fuzzy, undirected)

In [16] code · no output (function definitions)
def _camel_split(text): """AlbertEinstein → albert einstein""" spaced = re.sub(r"([a-z])([A-Z])", r"\1 \2", text) return spaced.lower() def _node_word_score(anchor_words, node): node_words = set(re.findall(r"[a-z]+", _camel_split(node))) return len(anchor_words & node_words) / max(len(anchor_words), len(node_words)) def search_graph(anchor, depth=2): if anchor not in G: # Fuzzy fallback: find closest node by word overlap anchor_words = set(w for w in re.findall(r"[a-z]+", _camel_split(anchor)) if len(w) > 2) best_node, best_score = None, 0.0 for node in G.nodes(): score = _node_word_score(anchor_words, node) if score > best_score: best_score, best_node = score, node if best_node and best_score >= 0.25: nonlocal_anchor = best_node # use the closest match undirected_G = G.to_undirected() # ignore arrow direction # BFS on undirected graph from anchor node ...
💡 Why CamelCase splitting?

The LLM sometimes wrote entity names without spaces — like AlbertEinstein or QuantumMechanics. If you searched for "Albert Einstein" but the graph only had "AlbertEinstein", a plain string match would fail.

_camel_split uses a regex to insert spaces before capital letters that follow lowercase letters:

AlbertEinstein → Albert Einstein → albert einstein ManhattanProject → Manhattan Project → manhattan project

This normalizes both the user's query words and the graph node names before comparing them.

💡 Why use undirected search?

When a directed graph has messy data, arrows often point in unexpected directions. For example, the LLM might extract (Albert, developed, Einstein) when it should be (Einstein, developed, Relativity). In directed BFS from "Einstein", you'd never reach "Albert" because the arrow goes the wrong way.

Converting to undirected (G.to_undirected()) treats all edges as bidirectional — much more resilient when extraction quality is imperfect.

🔵 Word-overlap scoring

Given anchor words {albert, einstein} and a graph node "Albert Einstein":

node_words = {albert, einstein} intersection = {albert, einstein} (2 words match) score = 2 / max(2, 2) = 1.0 ← perfect match For node "AlbertEinstein": _camel_split → "albert einstein" → {albert, einstein} score = 2/2 = 1.0 ← also perfect match! For node "Subject1": _camel_split → "subject1" → {subject} (digits stripped) score = 0/2 = 0.0 ← no match

📌 8. Anchor Detection (Cell 18)

In [18] code · no output (function definition)
def detect_anchor_entity(question): question_words = set(w for w in re.findall(r"[a-z]+", question.lower()) if len(w) > 2) # Score every known entity against the question words for entity in sorted_entities: normalized = camel_split(entity).lower() entity_words = re.findall(r"[a-z]+", normalized) score = sum(1 for w in entity_words if w in question_words) / len(entity_words) # score = fraction of entity words found in question return best_entity # entity with highest overlap score
🔵 What is an Anchor Entity?

The anchor is the "main character" of your question — the node in the graph where you start searching. If you ask "What did Galileo discover?", the anchor is Galileo. You then do BFS from that node to find all connected facts.

Getting the anchor right is crucial: a wrong anchor means you search the wrong neighborhood and get irrelevant facts.

⚠️ Why not use the LLM for anchor detection?

An earlier version did use the LLM here. It asked: "What is the main entity in this question?" But with qwen2:0.5b, the LLM would answer things like:

"The main entity is Subject3 (Albert Einstein), which represents..."

The small model couldn't reliably identify just the entity name. It added verbosity that broke the matching. The deterministic word-overlap approach is much more reliable — and requires no LLM call, making it faster.

🔵 What is Spring Layout?

nx.spring_layout uses a physics simulation: nodes repel each other like magnets, while edges act like springs pulling connected nodes together. After the simulation converges, nodes that are tightly connected cluster visually, and isolated nodes spread to the edges.

The seed=42 makes the layout deterministic — same input always produces the same visual arrangement. Without it, the graph would look different every run.

🤔 10. The Q&A Pipeline (Cell 21)

In [21] code · no output (function definition — the main RAG function)
def answer_question_with_graph(query, depth=2): # Step 1: Find the best matching entity in the graph anchor_entity = detect_anchor_entity(query) # Step 2: BFS to get the neighborhood subgraph subgraph = search_graph(anchor_entity, depth=depth) # Step 3: Convert subgraph edges to text strings context = [f"{u} {data['relation']} {v}." for u, v, data in subgraph.edges(data=True)] context_text = "\n".join(context) # Step 4: Ask LLM to answer using only the graph context response = llm.invoke( f"Answer the question based on these facts:\n\n{context_text}\n\nQuestion: {query}" ) return context, response.content
🔵 This is the RAG loop — explained step by step

This function is the heart of the whole notebook. Here's what happens end-to-end when you call it:

  1. Anchor detection: "How did Einstein contribute to the atomic bomb?" → detect_anchor_entity finds "Albert Einstein" (score 1.0 because both "albert" and "einstein" appear in the question).
  2. Graph retrieval: BFS from "Albert Einstein" collects all nodes within 2 hops in the undirected graph.
  3. Context formatting: Each edge becomes a sentence: "Albert Einstein discovered-theory-of-relativity Quantum Mechanics."
  4. LLM answering: The LLM receives: "Answer based on these facts: [graph edges as text] Question: [your question]" and generates a response.
💡 Why does GraphRAG help? (vs just asking the LLM directly)

If you ask the LLM directly: "How did Einstein contribute to the atomic bomb?" — it answers from training data. This works for famous facts, but fails for:

  • Private company data the LLM never saw
  • Recent events after the training cutoff
  • Specific facts in your documents

With GraphRAG, you inject your specific knowledge into the prompt. The LLM is guided to use your data, not just its memorized training data. The graph structure ensures only the relevant facts are included — not the entire knowledge base, which would overflow the context window.

Breaking down the output:

Why score 1.00?

The question contains both "albert" and "einstein". The entity "Albert Einstein" in the graph contains both "albert" and "einstein". Score = 2 matching words / 2 entity words = 1.0. Perfect match.

Why are the Retrieved Chunks so messy?

Those chunks come directly from the graph edges. They look like "Albert Einstein albert einstein Theoretical physicist.." because:

  • The LLM extracted "albert einstein" (lowercase) as a separate entity from "Albert Einstein"
  • It also created "Theoretical physicist" as a node connected to Einstein
  • These messy edges end up in the subgraph and are formatted as context text
💡 The "Lucky Answer" phenomenon

Despite the garbage context ("Albert Einstein albert einstein Theoretical physicist.."), the LLM's answer is actually quite good — it correctly explains Einstein's role in the Manhattan Project.

Why? Because the LLM already knows about Einstein and the atomic bomb from its training data. The graph context confirmed "Einstein → physics → bomb" direction, and the model filled in the rest from memory.

This would NOT work for private data. If the question were "How did CEO John Smith contribute to Project X?" and the graph context was garbage, the LLM would have no fallback and would hallucinate completely.

🐛 12. Why Everything Looks Weird — The Small Model Problem

🔴 The Core Issue: Model Size vs Task Complexity

qwen2:0.5b has 500 million parameters. Structured information extraction (producing clean JSON or triples from free text) typically requires at least 7–13 billion parameters to work reliably. The task involves:

  • Understanding what counts as a "named entity" vs a description
  • Recognizing full names as single units (Albert Einstein, not Albert + Einstein)
  • Following strict output format rules without deviating to natural language
  • Not copying example placeholder text from the prompt

This is simply too much for a 0.5B model. It's impressive it works at all.

What you got (qwen2:0.5b) What you'd get (GPT-4o or Llama-3-70b)
Nodes: "Einstein", "Albert", "Albert Einstein" (three separate nodes for one person) Single node: "Albert Einstein"
Nodes: "Subject1", "Predicate1", "Object1" (prompt contamination) Clean entity names: "Theory of Relativity", "Manhattan Project"
"significantly advancing radioactivity research." as a node (phrase dump) Separate nodes: "Radioactivity", "Research", with "advancing" as predicate
heliocentesis (misspelling) Heliocentrism (correct)
Messy retrieved chunks with duplicate entities Clean, precise facts: "Einstein → influenced → Manhattan Project"

🎓 14. Key Concepts Glossary

Knowledge Graph
A database that stores information as nodes (entities: people, places, concepts) and edges (relationships). Unlike a table, it's optimized for navigating connections. Example: Google's Knowledge Panel uses a knowledge graph to show you "Albert Einstein → spouse → Mileva Marić".
Triple (S, P, O)
The atomic unit of a knowledge graph: Subject → Predicate → Object. One fact. Every edge in the graph is a triple. Named after the Resource Description Framework (RDF) from the Semantic Web standards.
RAG
Retrieval Augmented Generation. A pattern where you retrieve relevant context from a knowledge store and inject it into the LLM's prompt, so the model can answer questions about data it wasn't trained on.
GraphRAG
RAG where the knowledge store is a graph instead of a vector database. Retrieval follows graph relationships (BFS) rather than vector similarity. Better for multi-hop reasoning: "Who influenced the person who discovered X?"
MultiDiGraph
A directed graph (arrows have direction) that allows multiple edges between the same two nodes. "Einstein influenced Quantum Mechanics" AND "Einstein contributed to Manhattan Project" can both exist as edges from Einstein.
BFS (Breadth-First Search)
A graph traversal algorithm that explores nodes level by level — first all direct neighbors (depth 1), then their neighbors (depth 2), etc. Used here to collect the "neighborhood" around an anchor entity.
Anchor Entity
The starting node for a graph search, determined by matching the user's question words to graph entity names. The quality of anchor detection directly affects the quality of retrieved context.
Temperature
An LLM parameter (0.0–1.0+) controlling output randomness. 0 = deterministic/repetitive. 0.7 = balanced. 1.0+ = creative but unpredictable. For structured extraction tasks, low temperature (0.0–0.2) gives more consistent formatting.
Chunking / Text Splitting
Breaking long documents into smaller pieces that fit within an LLM's context window. RecursiveCharacterTextSplitter tries natural break points (paragraphs → sentences → words) before cutting mid-word. Overlap ensures context isn't lost at boundaries.
LangChain LCEL
LangChain Expression Language. The | pipe syntax chains components into a pipeline: prompt | llm | parser. Each component's output becomes the next one's input. Makes complex AI pipelines readable and composable.
CamelCase Normalization
Converting camelCase or PascalCase strings to space-separated words: AlbertEinstein → Albert Einstein. Used here to handle LLM outputs that wrote names without spaces. The regex ([a-z])([A-Z]) finds the boundary between lower and upper case.
Spring Layout
A graph visualization algorithm that simulates physics: nodes repel each other (like magnets), edges attract connected nodes (like springs). The result is an organic, aesthetically pleasant layout where clusters of connected nodes appear visually close together.
Word-overlap scoring
A simple but effective similarity metric: count how many words two strings share, divided by the total unique words. score = |intersection| / |union| (Jaccard similarity). Used here to match question words to entity names.
Ollama
An open-source tool for running LLMs locally. Manages model downloads, serves them over HTTP (port 11434), handles GPU/CPU inference. Think of it as "Docker for LLMs" — a runtime that packages models and serves them via a standard API.
