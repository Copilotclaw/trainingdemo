📓 Notebook Walkthrough

What each group of cells does — and why.

Group 1 — Setup

Install Python packages and pick your backend

The first two cells handle everything you need before running any AI code:

  • Dependencies — installs openai (the API client), Pillow (image handling), and requests silently.
  • Backend config — set BACKEND = "ollama" for fully local inference (no data leaves your machine) or "dashscope" for the cloud API (no GPU needed).
  • Model selection — choose your Qwen 3.5 size here. Smaller = faster; larger = smarter.
# Two options — pick one:
BACKEND      = "ollama"         # local, private, free
BACKEND      = "dashscope"      # cloud, no GPU needed
OLLAMA_MODEL = "qwen3.5:1.5b"  # change size here
✅
Free Colab tier? Use "dashscope" — you don't need a GPU. Get a free API key at dashscope.aliyuncs.com.
⚙️

Group 2 — Client & Ollama Setup

Start the server (if local) and wire up the API client

This group does two things depending on your backend:

  • Ollama path — installs Ollama on Colab if missing, starts the server process in the background, and pulls the model you chose (only downloads once).
  • DashScope path — nothing to install, just skips to client creation.

Both backends then create the same OpenAI client object pointing at the right URL — so all subsequent code works identically regardless of backend.

Config → Start server → Pull model → OpenAI client → chat() helper

The chat() helper wraps the API call into a single line — pass a prompt, get a string back. Used by every demo cell below.

💬

Group 3 — Text Chat Playground

Talk to your model — change the prompt to anything

Two hands-on cells designed to be edited and re-run:

  • Simple prompt — change MY_PROMPT to any question. The model answers and prints its response tagged with the model name.
  • System role — the system message sets the model's persona before the conversation starts. Change PERSONA to give it any character: teacher, code reviewer, sarcastic chef, whatever.
# Swap in your own prompt
MY_PROMPT = "Explain quantum computing in 2 sentences."

# Or give the model a persona
PERSONA  = "You are a pirate who explains tech."
💡
Try the same prompt on the 0.8B and the 9B model — the quality difference is visible and educational.

Group 4 — Streaming

Watch tokens arrive in real-time

Instead of waiting for the full response, streaming mode sends each token (word fragment) as it's generated. This is how ChatGPT and Claude display responses while they think.

The cell uses stream=True and loops over chunk.choices[0].delta.content, printing each fragment immediately with flush=True.

stream = client.chat.completions.create(
    model=ACTIVE_MODEL,
    messages=[...],
    stream=True,
)
for chunk in stream:
    print(chunk.choices[0].delta.content, end="", flush=True)

This also shows why small models feel "fast" — a 1.5B model streaming at 80 tokens/sec feels instant even if total response time is the same as a 9B model generating all at once.

Same prompt, multiple sizes — see the difference

Group 6 — Vision Playground

Describe images with the same model that handles text

All Qwen 3.5 models are natively multimodal. The vision cells show two patterns:

  • Image from URL — send a public image URL directly in the message. No downloading, no base64. The model fetches and processes it.
  • Visual Q&A — a second cell with MY_IMAGE_URL and MY_QUESTION you can swap out freely. Ask anything about the image.

The message structure uses a content array — mixing an image_url part and a text part in the same user message:

# Vision message structure
messages=[{
    "role": "user",
    "content": [
        {"type": "image_url", "image_url": {"url": IMAGE_URL}},
        {"type": "text",      "text": "What's in this image?"},
    ],
}]
💡
This is the same OpenAI vision API format used by GPT-4o — so code that works here transfers directly to other providers.
📁

Group 7 — Vision with Local File

Upload your own image from disk

The previous vision cells use URLs. This cell shows how to send a local file — a screenshot, photo, or drawing from your machine.

  • In Colab — uses google.colab.files.upload() to open a file picker, then reads the bytes.
  • Local Jupyter — reads a file by path from disk (change filename = "test.jpg").

Either way, the bytes get base64-encoded and embedded directly in the message as a data: URL — no uploading to a third-party server.

# Base64-encode the image
b64      = base64.b64encode(image_bytes).decode()
data_url = f"data:image/jpeg;base64,{b64}"
🔒
When using Ollama (local backend), the image never leaves your machine — everything runs on-device.
