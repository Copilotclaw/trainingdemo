Dense Embedding Retrieval

Semantic search with bi-encoders + FAISS

Baseline technique
🚨 Problem it solves

Traditional keyword search fails when the user's phrasing doesn't match the document's wording. Ask "How do LLMs avoid hallucinations?" and you miss documents that say "grounding language models with external knowledge" — same concept, completely different words.

Dense embeddings solve this by converting text into vectors where semantic proximity = vector proximity. Two sentences with the same meaning cluster close together in vector space even if they share zero tokens.

How it works

A bi-encoder (e.g. all-MiniLM-L6-v2) encodes each document into a fixed-size vector at index time. At query time, the query is encoded with the same model. The similarity score is the cosine similarity — the dot product of L2-normalised vectors.

FAISS (Facebook AI Similarity Search) makes retrieval fast. IndexFlatIP does exact inner-product search. For millions of vectors, approximate variants (IVF, HNSW) trade a small accuracy loss for dramatically lower latency.

Index time (once): doc_1: "RAG fetches relevant docs..." -[encoder]-> [0.12, -0.34, 0.89, ...] -> FAISS doc_2: "BM25 uses term frequency..." -[encoder]-> [0.55, 0.21, 0.03, ...] -> FAISS doc_3: "Dense embeddings map text..." -[encoder]-> [-0.11, 0.78, 0.44, ...] -> FAISS Query time: query: "How does semantic retrieval work?" -[same encoder]-> [0.08, -0.29, 0.91, ...] | cosine similarity against all indexed vectors | top-k results score 0.94 -> doc_3 (closest in vector space) score 0.87 -> doc_1 score 0.72 -> doc_7
dense_retrieval.py
import numpy as np
from sentence_transformers import SentenceTransformer
import faiss

# ── 1. Load a lightweight bi-encoder ─────────────────────────────────
# all-MiniLM-L6-v2: 22M params, 384-dim output, ~30ms per sentence on CPU
embedder = SentenceTransformer("all-MiniLM-L6-v2")

# ── 2. Your document corpus ───────────────────────────────────────────
CORPUS = [
    "Retrieval-Augmented Generation (RAG) combines a retrieval system with a language model.",
    "Dense embeddings map text into a high-dimensional vector space.",
    "FAISS is a library for efficient similarity search over dense vectors.",
    "BM25 is a bag-of-words ranking function based on TF-IDF.",
    "HyDE asks the LLM to write a hypothetical answer and embeds that instead of the query.",
]

# ── 3. Encode all documents at index time (do this ONCE) ──────────────
# normalize_embeddings=True  =>  dot product == cosine similarity
doc_vectors = embedder.encode(
    CORPUS,
    normalize_embeddings=True,
    show_progress_bar=True
)
# doc_vectors.shape == (5, 384)

dim = doc_vectors.shape[1]                 # 384 for all-MiniLM-L6-v2
index = faiss.IndexFlatIP(dim)             # Inner Product on normalised vecs = cosine
index.add(doc_vectors.astype(np.float32))  # FAISS requires float32
print(f"Index contains {index.ntotal} vectors")

# ── 4. Search ─────────────────────────────────────────────────────────
def dense_search(query: str, k: int = 3) -> list[tuple[float, str]]:
    """Encode query, find top-k most similar documents."""
    q_vec = embedder.encode([query], normalize_embeddings=True)  # (1, 384)
    scores, indices = index.search(q_vec.astype(np.float32), k)
    return [(float(scores[0][i]), CORPUS[indices[0][i]]) for i in range(k)]

# Example
results = dense_search("How does semantic similarity search work?", k=3)
for score, doc in results:
    print(f"  [{score:.3f}]  {doc[:80]}")

✓ When to use

  • Queries use natural language, not keywords
  • Documents and queries may use different vocabulary
  • Cross-lingual retrieval (multilingual models)
  • Semantic similarity matters more than exact match

⚠ Limitations

  • Requires inference (slower than BM25)
  • Can miss exact rare terms (product codes, names)
  • Embedding quality varies by domain
  • Hard to debug: why was this doc retrieved?

BM25 — Sparse Keyword Retrieval

Probabilistic TF-IDF ranking, no neural network required

Fast · Interpretable
🚨 Problem it solves

Dense embedding models compress meaning into fixed-size vectors. This compression can lose rare, specific tokens — product codes, medical abbreviations, version numbers, proper nouns. Ask "Does this support CUDA 12.4?" and a general-purpose embedding model may surface GPU acceleration docs, missing the exact version match.

BM25 matches exact tokens with statistically-motivated weights. It remains state-of-the-art for keyword-heavy retrieval and is the backbone of most enterprise search engines (Elasticsearch, OpenSearch, Solr all default to BM25).

The BM25 formula

BM25 scores a document by summing per-term scores across all query terms present in the document:

BM25 score for query term t in document d: score(t, d) = IDF(t) x tf(t,d) * (k1 + 1) ---------------------------- tf(t,d) + k1*(1 - b + b*|d|/avgdl) IDF(t) = log[ (N - df(t) + 0.5) / (df(t) + 0.5) + 1 ] tf(t,d) = count of term t in document d |d| = length of document d in tokens avgdl = average document length across entire corpus N = total number of documents df(t) = number of documents containing term t k1 = 1.5 controls term-frequency saturation (diminishing returns) b = 0.75 controls length normalisation strength Final score = sum of score(t, d) for every query term t
bm25_retrieval.py
from rank_bm25 import BM25Okapi
import re

CORPUS = [
    "Retrieval-Augmented Generation (RAG) combines a retrieval system with a language model.",
    "BM25 is a bag-of-words ranking function based on the probabilistic model of information retrieval.",
    "Dense embeddings map text into a high-dimensional vector space.",
    "FAISS (Facebook AI Similarity Search) is a library for efficient similarity search over dense vectors.",
    "HyDE asks the LLM to write a hypothetical answer, embeds it, and uses that vector for retrieval.",
    "Cross-encoder re-rankers score each (query, document) pair jointly using full attention.",
]

# ── 1. Tokenise every document ────────────────────────────────────────
# Keep it simple: lowercase + remove punctuation + split on whitespace.
# In production, consider nltk.word_tokenize or spaCy for better stemming.
def tokenise(text: str) -> list[str]:
    return re.sub(r"[^\w\s]", "", text.lower()).split()

tokenised_corpus = [tokenise(doc) for doc in CORPUS]

# ── 2. Build the BM25 index ───────────────────────────────────────────
# BM25Okapi is the standard variant: pure Python, no GPU required,
# index builds in milliseconds even for millions of documents.
bm25 = BM25Okapi(tokenised_corpus, k1=1.5, b=0.75)

# ── 3. Query ─────────────────────────────────────────────────────────
def bm25_search(query: str, k: int = 3) -> list[tuple[float, str]]:
    """Tokenise query and return top-k (score, document) pairs."""
    q_tokens = tokenise(query)
    scores = bm25.get_scores(q_tokens)  # one score per document

    ranked = sorted(
        zip(scores, CORPUS),
        key=lambda x: x[0],
        reverse=True
    )
    return [(float(s), d) for s, d in ranked[:k]]

# Exact keyword match: BM25 wins here
results = bm25_search("BM25 term frequency inverse document frequency ranking", k=3)
for score, doc in results:
    print(f"  [{score:.3f}]  {doc[:80]}")

# Note: BM25 scores 0 for semantic paraphrases with no token overlap.
# "How does sparse retrieval compute relevance?" scores 0 because none
# of those tokens appear literally in the BM25 document above.
💡 Hybrid Search with Reciprocal Rank Fusion

The best production systems run both BM25 and dense retrieval and merge with RRF. For each document, the fused score is 1/(rank_bm25 + 60) + 1/(rank_dense + 60). Adding ranks (rather than raw scores) avoids scale mismatch and consistently outperforms either method alone. The constant 60 dampens the impact of very low ranks.

✓ When to use

  • Domain jargon, product codes, version strings
  • Expert users who know the exact terminology
  • No GPU available — pure CPU and RAM
  • Need interpretability (see which tokens matched)

⚠ Limitations

  • Zero score when no query tokens appear in doc
  • Fails on paraphrases and synonyms
  • Sensitive to tokenisation quality
  • Language-specific: does not handle multilingual

HyDE — Hypothetical Document Embeddings

Bridge the query-document vocabulary gap with generative search

LLM-augmented
🚨 Problem it solves

Users write questions. Your corpus contains answers. These live in different regions of embedding space — even with a good encoder. "What algorithm finds nearest neighbours in high dimensions?" embeds quite differently from "FAISS supports exact and approximate nearest-neighbour algorithms and scales to billions of vectors."

HyDE (Gao et al., 2022) bridges this query-document gap by having an LLM hallucinate a plausible answer, then embedding the hallucinated answer instead of the raw question. The generated passage lives in the same document-language space as your corpus — dramatically improving retrieval recall even when the LLM's answer contains factual errors.

How it works

Standard dense retrieval: user query -[encoder]-> query vector (question space) -[FAISS]-> top-k HyDE retrieval: user query -[LLM]-> hypothetical document -[encoder]-> hypo vector | (now in document space, not question space) | FAISS search | top-k real docs Key insight: the hypothetical doc does NOT need to be factually correct. Its job is to use document-like language that maps to the right region of vector space.
hyde_retrieval.py
import ollama
import numpy as np
from sentence_transformers import SentenceTransformer
import faiss

embedder = SentenceTransformer("all-MiniLM-L6-v2")
# Assume CORPUS, doc_vectors, faiss_index already built (see technique 1)

def llm(prompt: str, system: str = "") -> str:
    """Call the local Ollama LLM and return the response text."""
    messages = []
    if system:
        messages.append({"role": "system", "content": system})
    messages.append({"role": "user", "content": prompt})
    resp = ollama.chat(model="llama3.2", messages=messages)
    return resp["message"]["content"].strip()

def hyde_search(query: str, k: int = 3) -> list[tuple[float, str]]:
    """
    HyDE: generate a hypothetical document for the query,
    embed it, and retrieve using that embedding instead of the raw query.
    """
    # Step 1 -- Ask the LLM to write a hypothetical answer.
    # We want a passage that *looks like* a document in our corpus,
    # not a conversational Q&A answer.
    hyde_prompt = (
        f"Please write a short passage (2-3 sentences) that directly answers "
        f"the following question. Write as if this passage appears in a technical "
        f"reference document. Do NOT start with 'Sure' or repeat the question.

"
        f"Question: {query}"
    )
    hypothetical_doc = llm(hyde_prompt)
    print(f"Hypothetical doc generated:
  {hypothetical_doc}
")

    # Step 2 -- Embed the hypothetical DOCUMENT (not the original query!)
    # This vector lives in "answer space" rather than "question space",
    # aligning better with how real corpus documents were encoded.
    hypo_vec = embedder.encode([hypothetical_doc], normalize_embeddings=True)

    # Step 3 -- Search with the hypothetical embedding
    scores, idxs = faiss_index.search(hypo_vec.astype(np.float32), k)
    return [(float(scores[0][i]), CORPUS[idxs[0][i]]) for i in range(k)]

# Example: the query uses vague language; the corpus uses specific terms
query = "What technique uses a fake generated answer to improve search?"
results = hyde_search(query, k=3)
for score, doc in results:
    print(f"  [{score:.3f}]  {doc[:80]}")
⚡ Latency cost

HyDE adds one full LLM inference per query. With a local Llama 3.2 (3B) model, that's roughly 0.5–2 seconds. For latency-sensitive applications, cache hypothetical docs for common queries, generate them asynchronously, or batch-prefetch for predictable query patterns.

✓ When to use

  • Open-ended questions in natural language
  • Query vocabulary differs from document vocabulary
  • Technical docs with very specific terminology
  • Already using an LLM in the pipeline (marginal cost)

⚠ Limitations

  • Adds 1 LLM call latency per query
  • Confidently wrong hallucinations can derail retrieval
  • Works best with a capable LLM (small models generate poor hypotheticals)
  • Hard to debug: query → hypothetical → retrieved doc

MultiQuery Retrieval

Expand one query into N variants to dramatically improve recall

Recall booster
🚨 Problem it solves

A single query captures only one possible framing of the user's intent. "How do local language models help with private search?" might match well-worded corpus documents, but entirely miss documents about "on-premise LLM inference," "air-gapped retrieval pipelines," or "running models without cloud APIs." A single query vector is just one point in the embedding space — relevant documents in other neighbourhoods are invisible to it.

MultiQuery treats retrieval as a multi-perspective problem: generate several semantically diverse rephrases, retrieve for all of them, and merge. Recall improves because different phrasings reach different regions of the embedding space.

How it works

original query: "How do local models help with private search?" | +--[LLM paraphrase]-> variant 1: "Running LLMs locally for data privacy" +--[LLM paraphrase]-> variant 2: "On-premise language model inference pipelines" +--[LLM paraphrase]-> variant 3: "Air-gapped RAG without cloud dependencies" For each variant (+ original query): dense_search(v1) --> {doc_A:0.91, doc_C:0.87, doc_F:0.82} dense_search(v2) --> {doc_B:0.88, doc_C:0.85, doc_G:0.80} dense_search(v3) --> {doc_D:0.92, doc_A:0.83, doc_E:0.79} Merge: per doc, keep the BEST score seen across all variant retrievals Result: {doc_A, doc_B, doc_C, doc_D, doc_E, doc_F, doc_G} -- much wider coverage!
multiquery_retrieval.py
import ollama
import numpy as np
from sentence_transformers import SentenceTransformer
import faiss

embedder = SentenceTransformer("all-MiniLM-L6-v2")
# Assume CORPUS, faiss_index already built

def llm(prompt: str) -> str:
    resp = ollama.chat(model="llama3.2", messages=[{"role": "user", "content": prompt}])
    return resp["message"]["content"].strip()

def dense_search(query: str, k: int = 3) -> list[tuple[float, str]]:
    q_vec = embedder.encode([query], normalize_embeddings=True)
    scores, idxs = faiss_index.search(q_vec.astype(np.float32), k)
    return [(float(scores[0][i]), CORPUS[idxs[0][i]]) for i in range(k)]

def multiquery_search(
    query: str,
    n_variants: int = 3,
    k_per_variant: int = 3
) -> list[tuple[float, str]]:
    """
    MultiQuery retrieval:
      1. Generate n_variants rephrases of the original query via LLM.
      2. Run dense retrieval for each variant + the original.
      3. Merge: keep the best score seen for each unique document.
    """
    # Step 1 -- Generate query variants
    mq_prompt = (
        f"Generate {n_variants} different phrasings of the following search query. "
        f"Each phrasing should emphasise a slightly different aspect or use different "
        f"vocabulary. Output ONLY the {n_variants} queries, one per line, "
        f"no numbering, no explanations.

Original query: {query}"
    )
    variants_raw = llm(mq_prompt)
    variants = [v.strip() for v in variants_raw.strip().splitlines() if v.strip()]
    variants = variants[:n_variants]
    all_queries = [query] + variants

    print(f"Running {len(all_queries)} query variants:")
    for q in all_queries:
        print(f"  - {q}")

    # Step 2 -- Retrieve for every variant, track best score per document
    best_scores: dict[str, float] = {}  # doc_text -> best score seen
    for q in all_queries:
        for score, doc in dense_search(q, k=k_per_variant):
            if doc not in best_scores or score > best_scores[doc]:
                best_scores[doc] = score

    # Step 3 -- Sort by best score
    ranked = sorted(best_scores.items(), key=lambda x: x[1], reverse=True)
    return [(score, doc) for doc, score in ranked]

# Example
query = "How do local language models help with private search pipelines?"
results = multiquery_search(query, n_variants=3, k_per_variant=3)
for score, doc in results[:5]:
    print(f"  [{score:.3f}]  {doc[:80]}")
💡 Reciprocal Rank Fusion (RRF)

Instead of keeping the best score per document, you can fuse using RRF: for each retrieval run, score each document as 1 / (rank + 60), then sum across all runs. RRF is robust to score-scale differences between runs and is the standard for multi-retriever fusion in production. The constant 60 comes from empirical tuning and dampens the effect of very low ranks.

✓ When to use

  • Short or ambiguous user queries
  • Large, diverse corpus with many valid phrasings
  • Recall is more important than latency
  • Followed by a re-ranker to filter the larger candidate set

⚠ Limitations

  • 1 LLM call + N×dense_search calls per query
  • LLM variants may be poor paraphrases for very short queries
  • More candidates = higher downstream re-ranking cost
  • Results can be noisy if variants drift from original intent

Step-Back Prompting

Retrieve on an abstracted question to surface background context

Context enrichment
🚨 Problem it solves

Highly specific queries systematically miss important background context. "How does FAISS store vectors for fast lookup?" is so specific that it only retrieves FAISS-specific documents — but the LLM generating an answer might actually need foundational documents about vector spaces, approximate nearest-neighbour algorithms, or how cosine similarity works.

Step-Back prompting (Zheng et al., 2023) deliberately zooms out before retrieving. The LLM first rephrases the specific question as a broader, more abstract question. Retrieving on this broader question surfaces foundational context that narrow queries systematically miss, giving the downstream LLM richer material to reason from.

How it works

specific query: "How does FAISS store vectors for fast lookup at query time?" | +--[LLM step-back]-> "What are the principles behind vector indexing for similarity search?" | Retrieve on BOTH the step-back (abstract) and original (specific) queries: | dense_search(step_back_query) -> broad context docs doc: "Dense embeddings map text into high-dimensional vector space" doc: "FAISS supports exact and approximate nearest-neighbour algorithms" | dense_search(original_query) -> specific answer docs doc: "FAISS is a library for efficient similarity search over dense vectors" | Merge both sets -> rich context for LLM answer synthesis
stepback_retrieval.py
import ollama
from sentence_transformers import SentenceTransformer
import faiss
import numpy as np

embedder = SentenceTransformer("all-MiniLM-L6-v2")
# Assume CORPUS, faiss_index already built

def llm(prompt: str) -> str:
    resp = ollama.chat(model="llama3.2", messages=[{"role": "user", "content": prompt}])
    return resp["message"]["content"].strip()

def dense_search(query: str, k: int = 3) -> list[tuple[float, str]]:
    q_vec = embedder.encode([query], normalize_embeddings=True)
    scores, idxs = faiss_index.search(q_vec.astype(np.float32), k)
    return [(float(scores[0][i]), CORPUS[idxs[0][i]]) for i in range(k)]

def stepback_search(query: str, k: int = 3) -> list[tuple[float, str]]:
    """
    Step-Back retrieval:
      1. Ask the LLM to rephrase the specific question more abstractly.
      2. Retrieve on the abstract (step-back) question.
      3. Also retrieve on the original specific question.
      4. Merge both result sets, keeping best score per document.
    """
    # Step 1 -- Generate the abstract / step-back question
    sb_prompt = (
        f"Given the following specific question, produce a single more general, "
        f"abstract version that captures the broader concept or principle being asked about. "
        f"The generalised question should help retrieve background context that would "
        f"help answer the specific question. "
        f"Output ONLY the generalised question, nothing else.

"
        f"Specific question: {query}"
    )
    step_back_query = llm(sb_prompt)
    print(f"Step-back query: {step_back_query}
")

    # Step 2 -- Retrieve on the broader abstract query (background context)
    sb_results = dense_search(step_back_query, k=k)
    print("Background context (from step-back query):")
    for score, doc in sb_results:
        print(f"  [{score:.3f}]  {doc[:70]}")

    # Step 3 -- Retrieve on the original specific query (direct answer docs)
    orig_results = dense_search(query, k=k)
    print("
Specific docs (from original query):")
    for score, doc in orig_results:
        print(f"  [{score:.3f}]  {doc[:70]}")

    # Step 4 -- Merge, keeping best score per unique document
    best_scores: dict[str, float] = {}
    for score, doc in sb_results + orig_results:
        if doc not in best_scores or score > best_scores[doc]:
            best_scores[doc] = score

    merged = sorted(best_scores.items(), key=lambda x: x[1], reverse=True)
    return [(score, doc) for doc, score in merged]

# Example
query = "How does FAISS store vectors for fast lookup at query time?"
results = stepback_search(query, k=3)
print(f"
Final merged results ({len(results)} unique docs):")
for score, doc in results:
    print(f"  [{score:.3f}]  {doc[:80]}")

✓ When to use

  • Answering highly specific technical or factual questions
  • Corpus mixes overview docs with deep-dive details
  • Users need context to understand the specific answer
  • Multi-hop reasoning where background helps the LLM

⚠ Limitations

  • Step-back query may be too vague for narrow domains
  • Extra LLM call adds latency
  • Merging can surface off-topic background docs
  • Quality depends on LLM's abstraction capability

Cross-Encoder Re-Ranking

Two-stage pipeline: fast retrieval + accurate joint scoring

Precision booster
🚨 Problem it solves

Bi-encoders are fast because they encode query and document independently and compare the resulting vectors. But independence is also their weakness: the model never sees the query and document together, so it can't reason about specific word interactions, negations, or nuanced relevance signals.

A cross-encoder reads query and document jointly as a single concatenated input: [CLS] query [SEP] document [SEP]. Full transformer attention flows between every query token and every document token. This is far more accurate — but also far slower, scaling quadratically with combined input length.

The solution: use the fast bi-encoder to cheaply retrieve a candidate set (top 10–20 docs), then use the accurate cross-encoder to re-rank just those candidates. You get the speed of the bi-encoder and the precision of the cross-encoder.

How it works

Stage 1 -- Bi-encoder (fast, scales to millions of docs): query -[encoder]-> query_vec (computed at query time) doc_i -[encoder]-> doc_vec_i (pre-computed and cached at index time) cosine similarity -> retrieve top-20 candidates cheaply Stage 2 -- Cross-encoder (accurate, only runs on candidates): [CLS] query [SEP] doc_1 [SEP] -[transformer]-> relevance score: 0.94 [CLS] query [SEP] doc_2 [SEP] -[transformer]-> relevance score: 0.61 [CLS] query [SEP] doc_3 [SEP] -[transformer]-> relevance score: 0.88 ... [CLS] query [SEP] doc_20 [SEP] -[transformer]-> relevance score: 0.35 Full attention between query and doc tokens -> much more accurate Sort by cross-encoder score -> final top-k results
reranking.py
import numpy as np
from sentence_transformers import SentenceTransformer, CrossEncoder
import faiss

# ── Load both models ──────────────────────────────────────────────────
# Bi-encoder: fast, independent encoding for retrieval
embedder = SentenceTransformer("all-MiniLM-L6-v2")

# Cross-encoder: accurate, joint encoding for re-ranking.
# ms-marco-MiniLM-L-6-v2 is trained on the Microsoft MARCO passage ranking dataset.
# It outputs a raw relevance logit (higher = more relevant).
cross_encoder = CrossEncoder("cross-encoder/ms-marco-MiniLM-L-6-v2")

# Assume CORPUS, faiss_index already built

def dense_search(query: str, k: int = 10) -> list[tuple[float, str]]:
    q_vec = embedder.encode([query], normalize_embeddings=True)
    scores, idxs = faiss_index.search(q_vec.astype(np.float32), k)
    return [(float(scores[0][i]), CORPUS[idxs[0][i]]) for i in range(k)]

def rerank_search(
    query: str,
    candidate_k: int = 10,
    final_k: int = 3
) -> list[tuple[float, str]]:
    """
    Two-stage retrieval + re-ranking pipeline.

    Stage 1: Cast a wide net with the fast bi-encoder (candidate_k docs).
    Stage 2: Re-score each candidate jointly with the cross-encoder.
             The cross-encoder sees both the query AND the document text,
             enabling it to catch fine-grained relevance signals.
    """
    # Stage 1 -- Retrieve a wider candidate pool
    candidates = dense_search(query, k=candidate_k)
    candidate_docs = [doc for _, doc in candidates]
    print(f"Stage 1: {len(candidate_docs)} candidates from bi-encoder")

    # Stage 2 -- Cross-encoder scores every (query, doc) pair jointly.
    # Input must be a list of [query, document] pairs.
    pairs = [[query, doc] for doc in candidate_docs]
    ce_scores = cross_encoder.predict(pairs)  # shape: (candidate_k,)
    # ce_scores contains raw logits -- higher means more relevant

    # Re-sort by cross-encoder score
    reranked = sorted(
        zip(ce_scores.tolist(), candidate_docs),
        key=lambda x: x[0],
        reverse=True
    )

    print(f"
Stage 2: re-ranked by cross-encoder (showing top {final_k}):")
    for ce_score, doc in reranked[:final_k]:
        print(f"  [CE {ce_score:.3f}]  {doc[:70]}")

    return [(float(score), doc) for score, doc in reranked[:final_k]]

# Example
query = "What methods improve the accuracy of retrieved documents in RAG?"
results = rerank_search(query, candidate_k=10, final_k=3)
💡 Why not use the cross-encoder for all documents?

A cross-encoder must process every (query, document) pair at query time. With 100,000 documents, that's 100,000 full forward passes per query — completely infeasible for real-time applications. The bi-encoder pre-computes all document embeddings once and retrieves in milliseconds via FAISS. The two-stage design gives you speed and precision: the bi-encoder narrows the search space, the cross-encoder perfects the ranking.

✓ When to use

  • Precision matters (docs are passed to a costly LLM)
  • After MultiQuery: large merged set needs filtering
  • High-stakes domains: medical, legal, financial
  • When bi-encoder top-1 "feels wrong" but the right doc is in top-10

⚠ Limitations

  • ~50–200ms latency for 10–20 candidates on CPU
  • Cannot scale to full corpus — must pair with fast retrieval
  • ms-marco model trained on web queries; may need domain fine-tuning
  • Still not fully interpretable — just better scores

# Technique Core idea Speed LLM call? Best for Combine with
1 Dense Embedding Bi-encoder vectors + FAISS cosine search Fast No Semantic queries, multilingual BM25, Re-Ranking
2 BM25 TF-IDF keyword scoring, no neural net Fastest No Exact terms, jargon, no GPU Dense (hybrid RRF)
3 HyDE LLM generates hypothetical answer; embed that Medium Yes (1×) Vocabulary mismatch, open-ended questions Dense, Re-Ranking
4 MultiQuery LLM rephrases query N ways; merge all results Medium Yes (1×) Short/ambiguous queries, high recall Re-Ranking (filter merged set)
5 Step-Back LLM abstracts query; retrieve broad + specific Medium Yes (1×) Specific questions needing background context Dense, MultiQuery
6 Cross-Encoder Re-Rank Joint query+doc scoring via full transformer attention Slow (candidates only) No (separate model) Maximising precision, final stage of any pipeline Any of the above
🏗 Recommended production pipeline

For most real-world RAG systems, this stack covers 90% of use cases:

  1. Hybrid retrieval — Dense + BM25 with RRF (captures both semantic and keyword signals)
  2. MultiQuery or HyDE — for queries where recall is critical or vocabulary gap is known
  3. Cross-Encoder Re-Ranking — always last, on the merged candidate pool, before passing to the LLM
